Configuration

Mayfly reads your existing .ddev/config.yaml with no modifications required. Additional options let you customise how environments are named and what runs on deploy.

How configuration works

When you run preview-deploy, Mayfly packages your project's ddev configuration and sends it to the Mayfly host. Your existing .ddev/ folder is used as-is — any PHP version, database, add-ons, or environment variables you have configured locally will apply to the preview environment too.

Note: You do not need to change your .ddev/config.yaml to use Mayfly. Everything you already have configured will carry over.

Environment naming

Each environment is identified by a slug derived from your project name and branch:

text
{your-username}.{project-name}-{branch-slug}.mayfly.live

Branches with slashes (e.g. feature/my-thing) are normalised to hyphens automatically.

Mayfly config file

You can optionally add a .mayfly.yml file to your project root to control deploy behaviour:

yaml
# .mayfly.yml
post_deploy:
  - ddev exec "wp search-replace localhost https://"
ttl: 72h

Available options

post_deploy
A list of shell commands to run inside the environment after it has started. Use this for database imports, search-replace operations, cache clears, etc.
ttl
How long the environment lives before being automatically destroyed. Accepts durations like 24h, 72h, 7d. Defaults to 48h.

SSH keys

Authentication is handled entirely via SSH keys. Mayfly does not use passwords. During initial setup, your public key is registered against your account. All environments you deploy are accessible only to holders of the corresponding private key.

To add a second key (e.g. for a CI runner), contact us via Slack.
